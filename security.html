RTOS Security Design Principles

🏰

Least Privilege

Each task should have only the permissions it needs. Use the MPU (Memory Protection Unit) to restrict task access to its own stack and data regions. Prevent tasks from accidentally (or maliciously) accessing other task memory.

🔏

Synchronisation

Protect all shared data with appropriate synchronisation primitives. Use mutexes (not binary semaphores) where priority inheritance is needed. Keep critical sections as short as possible to minimise blocking time.

📏

Bounds Checking

Validate all input sizes before writing to buffers, queues, or stacks. Use static analysis (e.g., MISRA-C, PC-Lint) and runtime checks (stack canaries, watchdog timers) to catch violations early.

⏰

Timing Safety

Use sufficiently wide integer types (uint32_t or larger) for all timing variables. Always perform wrap-around-safe arithmetic using unsigned subtraction. Verify WCET with static analysis tools before deployment.

👁️

Monitoring

Instrument the system with a hardware watchdog, stack high-watermark checks, and deadline-miss hooks. Log anomalies to non-volatile storage for post-mortem analysis. Never silently swallow errors.

🧪

Testing

Test concurrent scenarios deliberately: inject task delays, flip scheduling order, stress test with maximum load. Use formal methods (model checking, timing analysis tools) for safety-critical systems.
